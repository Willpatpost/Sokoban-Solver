Sokomind improvements and implementation status
===============================================

Status reviewed against the repository after Improvement Sprint 11: productivity-
aware strategic child ordering in persistent exact search.

Standing maintenance rules
--------------------------

- After every change to solver behavior, search strategy, pruning, state
  representation, worker protocol, or solver orchestration, increment the
  solver version ID and keep every browser cache-busting reference synchronized.
- At the end of every sprint, review this file against the repository, update
  the sprint/status heading, move completed work to the appropriate section,
  and record any newly discovered limitations or follow-up work.

The original item numbers and priority order are retained for traceability.
"Done" means the stated outcome is implemented and adequately validated.
"Strong, bounded" means the implementation is useful and well tested within a
deliberate scope, but does not yet provide the complete capability originally
described. "Partial" means meaningful foundations exist, but important parts of
the stated outcome remain. Items in the final section have not been implemented
as backlog capabilities; incidental overlap does not count as delivery.

STRONG IMPLEMENTATION, WITH BOUNDED LIMITATIONS
-----------------------------------------------

3. Add exhaustive differential tests for every hard pruning rule.
   Implemented:
   - A reference BFS checks every current hard pruning rule independently over
     reachable states, including their combined successor behavior.
   - The harness exhaustively enumerates all 780 box, goal, and robot placements
     across every connected 2x3 floor layout. It independently identifies and
     checks the 60 solvable configurations alongside four larger authored boards.
   - It scans the complete corpus and retains the globally smallest false
     positive by floor size, box count, board area, and state size for reporting.
   Bounded limitations:
   - Exhaustive generation does not yet cover larger floors, multiple boxes, or
     typed-label combinations.
   - Counterexamples are selected from the corpus but not structurally minimized.
   - Coverage must continue to be extended whenever hard rejection rules change.
   Remaining validation expansion:
   - Add exhaustive or systematically sampled 3x3 and 3x4 floor layouts.
   - Cover multiple typed boxes, interacting doorways, corrals, and closed
     diagonals as those pruning rules are introduced.
   - Add property-based state generation and automatic counterexample shrinking
     so failures produce the smallest practical reproducible puzzle.

5. Add hidden or generated benchmark families.
   Implemented:
   - Generated mirror, rotation, label permutation, premature-goal, typed
     doorway-import, exact room-packing, and corral-reopening cases.
   - A separate procedural lane-warehouse family covers three and four boxes,
     longer push distances, and distinct typed boxes without deriving its boards
     from the built-in level catalog or the authored strategic seeds.
   - An independent exact A* over canonical push states certifies every procedural
     multi-box case with an admissible label-aware assignment bound.
   - The replay-validated validation suite now contains 16 deterministic cases.
   Bounded limitations:
   - The procedural family currently uses open lane geometry rather than shared
     bottlenecks, interacting staging capacity, or coupled packing orders.
   - The suite remains repository-visible and deterministic, not a secret test
     corpus; more separately maintained challenge layouts would further reduce
     tuning against familiar cases.

7. Add detailed solver performance instrumentation.
   Implemented:
   - Counters and timings cover reachability, matching, hashing, successor
     generation, pruning, caches, frontier behavior, and strategic reasoning.
   - Capability-detected heap telemetry records current, peak, delta, and sample
     counts. Node benchmarks inject a reliable heap sampler; unsupported browser
     workers report that status explicitly instead of returning false zeroes.
   - Isolated benchmark cases record worker loading, search, validation, first
     output, result delivery, shutdown, total process time, and heap deltas.
   - Browser portfolio logs expose first-message, wall-clock, termination-call,
     heap, and cancelled-worker measurements.
   Bounded limitations:
   - Per-allocation profiling and exact GC event/pause counts are not portable to
     ordinary browser workers; the runner reports explicit-GC availability but
     does not force collection during scored searches.
   - The specialized bidirectional director has less lifecycle detail than the
     standard portfolio path, although worker-owned heap telemetry is shared.

8. Replace repeated single-box searches with a compiled push graph.
   Implemented:
   - Static graph nodes encode a box cell and robot-region representative.
   - Transitions and player-aware distances are compiled, cached, and tested
     against the reference implementation.
   Bounded limitations:
   - The graph is geometry-based rather than intrinsically type-aware.
   - It searches forward from box starts instead of reverse-searching from each
     typed goal into reusable typed feasibility tables.

9. Replace string-heavy hot-path state data with dense representations.
   Implemented:
   - Integer cell IDs, dense neighbor/support tables, typed reachability arrays,
     compact parent directions, and deferred walking-path reconstruction.
   - Typed box layouts are encoded as sorted dense cell/label tokens and packed
     into collision-free BigInt identities, cached by immutable box-array object.
   - Exact robot cells or canonical reachable-region IDs are packed into the same
     identity for beam, bounded DFS, IDA*, bidirectional, and baseline search
     transposition maps. Diversity and exact-shard hashing consume the packed
     value directly instead of reconstructing a string.
   - Exhaustive small-board tests verify permutation invariance, typed-layout
     uniqueness, exact robot distinction, and canonical robot-region equivalence.
   Bounded limitations:
   - Coordinate-heavy solver logic still stores boxes as nested JavaScript arrays
     and clones those arrays for successors; the packed form is currently the
     state identity, not the sole state representation.
   - Strategic memo tables and cross-worker IDs retain compact strings where they
     simplify interoperability and diagnostics.
   - Box bitsets and truly incremental move-updated hashing are not implemented.

10. Reuse immutable board analysis across web workers.
    Implemented:
    - Prepared board seeds share topology, push distances, compiled graphs, dense
      board data, and other immutable analysis through structured cloning.
    - Completed bounded room-pattern and pair-conflict tables are included in the
      planning seed so subsequent workers do not repeat reverse enumeration.
    - Mutable caches remain worker-private.
    Bounded limitations:
    - Structured cloning still duplicates data in worker memory.
    - SharedArrayBuffer is intentionally avoided because the current GitHub Pages
      deployment is not cross-origin isolated.
    - Dynamic deadlock, heuristic, and local-search caches remain worker-private.

13. Add a goal-commitment safety oracle.
    Implemented:
    - Goal boxes are classified as temporary, conditionally safe, or proven using
      residual perfect matching, room dependencies, static immovability, dynamic
      support demand, typed doorway flow, and bounded exact local analyses.
    - State-complete evidence is cached by robot region and box signature. It can
      downgrade placements that occupy required support or staging cells and can
      prove movable boxes packed inside an exactly completed one-entry room.
    - Beam, bounded DFS, and IDA search treat proven boxes as walls. Forced,
      straight, and sequence macros re-evaluate commitment after every push.
    - The lock is differentially checked over the exhaustive tiny corpus and all
      252 pushes of the known Huge solution; it can be disabled diagnostically.
    Bounded limitations:
    - Movable proof is currently limited to completed bounded rooms; oversized
      rooms and exact corrals cannot promote boxes to proven.
    - Support dependencies are not yet assignment-aware or recursively proven.
    - Baseline push A* and gameplay retain ordinary legal movement semantics.

15. Add cached exact local searches for small rooms and corrals.
    Implemented:
    - Bounded exact room and corral searches are cached by local configuration.
    - They reason about packing, import/export requirements, doorway restoration,
      boundary viability, and reopening robot access.
    Bounded limitations:
    - Exactness applies only inside the selected local abstraction.
    - Rooms, domains, or box sets over conservative size limits are skipped.
    - Some large built-in puzzle regions therefore receive no local exact result.
    - Corral conclusions are currently used conservatively as ordering signals.

39. Keep expensive discovery workers live and recover a silent search lane.
    Strong, bounded implementation in solver build 2026-07-22.20:
    - Guided beam and bounded DFS searches report progress after either a bounded
      number of states or five seconds, preventing their state thresholds from
      exceeding the watchdog window on expensive puzzles.
    - Macro workers use a 250-state reporting interval even though their primary
      budget is only 4,000 states; anytime workers retain coarse state logging but
      gain elapsed-time liveness reports.
    - A watchdog-terminated guided beam is retried once from the same generic state
      with a reduced width and budget, sequence macros disabled, and nested
      continuation/endgame probes disabled. A second failure is final, preventing
      an infinite restart cycle.
    - Required-work accounting includes required recovery workers, so exact search
      cannot launch merely because an opening phase was replaced.
    - Regression tests cover elapsed-time progress below the state threshold and
      the bounded director recovery policy.
    Bounded limitations:
    - JavaScript cannot preempt a single synchronous state expansion; an individual
      expansion lasting the entire watchdog window can still be replaced.
    - A fresh Huge browser run is still required to confirm that the evacuation
      checkpoint now reaches packing and landmark-bridge consumers in production.

40. Reduce exact transposition churn without multiplying browser memory use.
    Strong, bounded implementation in solver build 2026-07-22.20:
    - Exact workers divide a device-memory-aware total transposition budget across
      active shards instead of always receiving 160,000 entries each.
    - A lone shard receives up to 320,000 entries, directly addressing the observed
      saturation and eviction slowdown, while parallel shards remain collectively
      bounded and low-memory devices receive a smaller total budget.
    - The selected per-shard capacity is written to the search log and deterministic
      policy tests cover low-memory, default, and multi-shard allocations.
    Bounded limitations:
    - Browser device-memory reporting is coarse and sometimes unavailable; the
      default is conservative rather than based on live heap pressure.
    - The table still uses bounded insertion-order replacement and does not resize
      during a contour.

43. Keep available Ultimate worker capacity productively occupied.
    Strong, bounded implementation in solver build 2026-07-22.21:
    - Queued direct searches can use hardware slots released by bounded forward and
      reverse workers while the evacuation search remains active; evacuation no
      longer freezes the entire direct queue.
    - Direct capacity follows remaining reported hardware capacity instead of a
      fixed two-worker ceiling, while the existing one-active-bridge restriction
      prevents landmark probes from monopolizing every lane.
    - Once persistent exact search begins, completion of a bridge or anytime worker
      triggers checkpoint-guided refill without interrupting or repartitioning the
      exact contour.
    - Checkpoints receive at most two anytime attempts. Four rotating strategic
      profiles and attempt-specific seeds provide bounded diversity, and newly
      published replayable checkpoints can feed later refill rounds.
    - Refill logs its active/capacity counts, and scheduler regression checks cover
      evacuation concurrency, retry bounds, exact-phase refill, and synchronized
      browser build identifiers.
    Bounded limitations:
    - Refill depends on eligible replayable checkpoints. The scheduler intentionally
      leaves a slot idle when only exhausted or over-attempted work remains.
    - Concurrent direct workers can increase CPU contention and memory pressure;
      the Huge browser trace should be compared with build 20 before increasing
      per-worker search budgets.
    - The current browser tests validate policy and wiring but do not yet provide
      the real-browser campaign coverage tracked by item 42.

41. Make expensive strategic ordering analyses productivity-aware.
    Strong, bounded implementation in solver build 2026-07-22.22:
    - Persistent exact search now separates parent commitment evidence used for
      proven box locking from child commitment and doorway signals used only for
      successor ordering. Every parent still receives the complete locking oracle.
    - Child strategic analysis is never evaluated for successors already outside
      the admissible contour, eliminating work that cannot affect that iteration.
    - A deterministic productivity gate evaluates 64 eligible branching states and
      compares baseline and enriched successor order. If the richer signals never
      change the order, it skips 512 eligible states and then samples again.
    - Telemetry reports strategic ordering evaluations, gated skips, and actual
      ordering changes in progress and completion logs.
    - A 10,000-state Huge probe exercised 1,920 evaluated states, 765 gated skips,
      and 311 genuine ordering changes; exact-path, shard, pruning-differential,
      saved-solution, smoke, and validation tests remain green.
    Bounded limitations:
    - Parent commitment, dependency, local-room, and doorway evidence remains active
      because it can prove safe box locks or supply multiple existing ordering terms.
    - The gate measures whether ordering changes, not whether a change ultimately
      reduces time to a solution; reviewed Huge traces remain necessary.
    - Guided beam searches retain full strategic scoring because their primary job
      is first-solution discovery rather than exhaustive contour proof.

17. Add reverse-proven goal-room packing tables that identify safe packing orders
    and reject room arrangements that cannot be completed from the doorway.
    Delivered so far:
    - Bounded one-entrance rooms compile a reusable reverse push-state table from
      the fully packed, label-correct goal arrangement.
    - Reverse states retain canonical robot-reachability regions, exact typed box
      layouts, remaining push distance, and every equally optimal next push.
    - Exact local-room analysis consults the table before repeating its forward

    - Tables are puzzle-independent and bounded to rooms of at most 16 cells,

      Oversized and cutoff cases retain the existing forward-search fallback.
    - A missing entry is only a proof about the bounded doorway abstraction. It
      remains diagnostic/ordering evidence and is not promoted to global hard
      pruning because a box may need to travel farther outside that abstraction.
    Remaining:
    - Prove when leaving and re-entering a one-entry room cannot help, allowing a
      complete table miss to become a safe hard rejection in those cases.
    - Extend table construction to larger rooms with compact state storage,
      decomposed subrooms, or on-disk/generated pattern data.
    - Share compiled immutable tables across workers once construction cost and
      structured-clone size are shown to be beneficial.

18. Add closed-diagonal deadlock detection and more complete freeze detection.
    Delivered so far:
    - Browser and Python searches reject a conservative two-row wall-ended
      diagonal when each open diagonal square is flanked by exactly one box and
      one wall, both boxes face outward toward their wall endpoint, and no square
      in the pattern offers a goal escape.
    - The detector is board-derived and contains no level names, coordinates, or
      saved-solution knowledge.
    - Independent unpruned state graphs enumerate every two-box/two-wall border
      arrangement and both mirror orientations. Only the two outward-facing cases
      are accepted, and both are exhaustively proven unsolvable.
    - The generated family exposed two tempting but solvable false-positive forms
      during development; they remain rejected by the final detector. A goal-escape
      variant is also proven solvable and participates in the combined hard-pruning
      differential suite.
    - The known Huge replay, generated validation families, and all existing hard
      pruning differentials remain intact under solver build 2026-07-22.13.
    Remaining:
    - Recognize box-ended diagonals and safe goal/wall sequences without losing
      valid inward-opening pushes.
    - Add broader transformed, typed, multi-box, and generated diagonal families.
    - Extend recursive freeze detection beyond the current conservative local
      component rule, promoting new cases only after differential validation.

19. Add small deadlock pattern databases and use exhaustive local enumeration to

    - A novel typed corridor-order conflict is detected where individual static

      prune counts. The known Huge replay remains valid after static-window caching

    Remaining:
    - Generate broader wall-pattern families and transformed typed cases rather

    - Add generic-box patterns that improve on existing 2x2/freeze rules without

20. Build pattern databases for the lower chamber and other small, strategically
    important rooms. Use exact local costs or dependency-aware conflict bounds to
    capture interaction and ordering costs missing from assignment distance.
    Delivered so far:
    - One-entrance goal rooms compile reverse multi-box push tables for two to four
      boxes whose complete typed goal sets belong to that room. A room may select
      an exclusive typed subset even when shared generic goals are also present.
    - The abstraction retains the full board's walls, labels, box collisions, and
      required support squares while removing unrelated boxes and granting the
      robot access everywhere. Its discovered distances are therefore admissible
      lower bounds on pushes by those labels.
    - The heuristic replaces the selected labels' Hungarian contribution with the
      stronger pattern value. It does not add both costs, and a label-disjoint
      combination prevents overlapping room/label patterns from being counted
      twice.
    - Tables are capped at 12,000 states. Distances discovered by reverse BFS stay
      exact after a cutoff; a missing entry simply retains ordinary assignment.
    - A typed interaction fixture independently proves the exact cost increase
      from Hungarian 1 to pattern 3. The 250-push Huge replay checks admissibility
      at every push; its lower chamber automatically selects C/D and rejects no
      valid state. No level name, saved route, or fixed coordinate enters search.
    - Build 2026-07-22.17 reports table builds, states, hits, and cumulative bound
      improvement. Prepared-board seeds reuse completed tables across workers.
    Remaining:
    - Support shared-label/generic goal subsets by minimizing over every valid
      candidate-box selection before using a cost in the lower bound.
    - Extend beyond four boxes with dense encodings, symmetry reduction, selective
      partitions, or offline generation without making worker startup excessive.
    - Add generated interacting-room benchmark families that measure bound gains
      and state reductions rather than only correctness and replay admissibility.
    - Port the proven bounded subset to Python after the browser representation and

36. Incrementally update exact box-to-goal assignments after one box moves.
    Delivered so far:
    - Browser successor layouts carry a weak parent hint identifying the single
      changed box row. Cached label assignments reuse every unchanged distance
      row and repair the Hungarian solution with one augmenting path.
    - Repair activates for label groups of at least five boxes; profiling showed
      that full matching is faster and leaner for the common smaller groups.
    - The repair is O(n^2) worst case rather than the proposed O(n); it remains
      an improvement over rebuilding the O(n^3) assignment and returns exactly
      the same admissible value.
    - Deterministic randomized tests compare 840 one-row repairs, including
      unreachable edges, against full Hungarian recomputation.
    - Telemetry counts incremental calls, fallbacks, and reused matrix rows.
      Solver build 2026-07-22.15 exposes these counters.
    Bounded limitations:
    - The optimization currently applies to the browser solver; Python still
      recomputes each label's assignment after a cache miss.
    - Layouts without a parent assignment safely use the full solver.
    - Assignment details are weakly keyed by the box-array object and structurally
      share unchanged rows, so they follow search-state lifetime rather than
      accumulating in a separate strong cache.
