Sokomind improvement campaign and sprint plan
==============================================

Plan reorganized after Improvement Sprint 13 and updated after Sprint 15 on
2026-07-23.

Mission
-------

Build a general-purpose Sokoban search that can find a solution whenever one
exists and can eventually prove failure when none exists, subject only to an
explicit user stop or reported platform/resource failure. Production search must
derive every decision from the supplied board and state. Built-in puzzle names,
coordinates, saved routes, and the saved Huge solution must never influence
production search. The Huge route remains diagnostic evidence only.


Campaign operating rules
------------------------

- Future sprints contain 3-5 related improvements and are listed below in
  dependency and priority order.
- Once a sprint begins, implement every item in it without waiting for separate
  "ready for next sprint" messages. Stop only for a genuine external blocker,
  required user authority, or evidence that cannot be obtained locally.
- A campaign request may authorize several consecutive sprints. Continue through
  the authorized range, maintaining versions and this file internally, and return
  one consolidated handoff instead of requesting approval between sprints.
- Aim to move every sprint item to Done. "Strong, bounded" is acceptable only
  when the full claim would be unsound, platform-impossible, or unsupported by
  proof after the planned work is exhausted. Record the exact remaining boundary;
  do not use the label merely to end a sprint early.
- Do not silently move unfinished work to another sprint. If an item misses its
  finish gate, keep the sprint open unless a documented hard boundary justifies
  Strong, bounded status.
- After every change to solver behavior, search strategy, pruning, state
  representation, worker protocol, or solver orchestration, increment the solver
  version ID and synchronize every browser cache-busting reference.
- After every sprint, update this file with delivered behavior, tests, benchmark
  results, new limitations, and the next sprint's dependency assumptions.
- Preserve completeness in exact search. Ordering and bounded discovery may use
  experimental signals, but pruning requires a proof and independent differential
  validation.
- Run the complete JavaScript and Python suites, saved-solution verification,
  smoke benchmark, and validation benchmark after every sprint. Run browser and
  Huge campaign gates whenever the sprint changes orchestration or browser flow.
- Prefer general generated fixtures over built-in-level assertions. Any useful
  pattern learned from the Huge route must be reformulated as board-derived logic
  and tested on unrelated puzzles.


Status vocabulary
-----------------

Done
  The intended practical outcome is implemented and adequately validated.

Strong, bounded
  The useful and safe portion is implemented, but a precisely documented proof,
  platform, or scope boundary remains.

Partial
  Important foundations exist, but the improvement has not reached its finish
  gate.

Backlog
  The named capability has not yet been implemented.

Sprint 14 - Exact-search contract and interruption resilience
-------------------------------------------------------------

Priority reason:
The campaign currently tests individual pruning rules and selected exact contours,
but the mission depends on the complete composition of canonicalization,
transposition dominance, sharding, bounds, termination, and result reporting.
These contracts must be proven before adding more pruning or orchestration.

45. Differentially certify the complete exact-search kernel.
    Status: Done in build 2026-07-23.25.
    Delivered:
    - Browser Push A* is compared with an independent unpruned step-state 0-1 BFS
      over all 120 distinct box, goal, and robot placements on a connected 2x3
      floor plus generic and typed multi-box boards.
    - Browser exact IDA* is checked against the same oracle with commitment locking,
      forced macros, and transposition capacities varied independently. Tests compare
      solvability and optimal push count rather than route survival alone.
    - Three-way shard unions, finite invalid incumbents, unbounded reruns,
      transposition eviction, default finite bounds, and canonical push behavior
      have explicit regression coverage.
    - Python Push A* is compared with a separate unpruned step-state 0-1 BFS over
      generic, typed, solvable, and unsolvable multi-box cases.
    - The new exact-kernel differential suite is a required GitHub Actions gate.

46. Define one trustworthy terminal-result contract.
    Status: Done in build 2026-07-23.25.
    Delivered:
    - Python, browser search, Web Worker protocol, director, GUI, CLI, benchmark
      results, and structured logs use solved, proven-unsolvable, cutoff, cancelled,
      and failed with an explicit reason.
    - Python's SearchResult remains tuple-unpack compatible while exposing status,
      reason, and cutoff fields to new callers.
    - Finite contours, guided portfolios, and bounded beams cannot claim
      unsolvability. Only certified unbounded exact exhaustion can do so.
    - Both production solvers independently replay and trim candidate solutions
      before reporting solved. Illegal, incomplete, or state-mismatched paths become
      failed results.
    - Worker exceptions become structured failures, repeated proof-worker crashes
      have a bounded recovery policy, user stops log cancellation, and existing
      stale-message guards remain covered.

47. Make persistent exact search stack-safe, resumable, and crash-recoverable.
    Status: Done in build 2026-07-23.25.
    Delivered:
    - Exact IDA* now uses an explicit traversal stack. A 600-push regression proves
      deep paths no longer depend on the JavaScript call stack.
    - Production exact workers yield after bounded state slices. Versioned
      checkpoints include a board/state hash, build, bound, shard, contour,
      traversal stack, counters, and a bounded recent transposition tail.
    - Transpositions remain a performance cache: omitted entries can only repeat
      work, while the serialized traversal stack preserves proof progress.
    - Compatible checkpoints persist in browser storage and resume after worker
      replacement or page reload. Board, build, bound, and shard mismatches are
      rejected, and Clear Saved Search explicitly removes the current proof state.
    - Deterministic tests interrupt at five different slice sizes, JSON-round-trip
      every checkpoint, and require the resumed outcome and push count to match an
      uninterrupted run. Storage round trips and incompatible checkpoints are
      independently tested.

Sprint completion gate:
Completed in build 2026-07-23.25. All three items are Done; exact search has an
independently checked completeness contract, every terminal outcome is
unambiguous, and interruption cannot silently turn a proof run into a restart,
false failure, or false proof.

Sprint 24 - User-authored puzzle workflow
-----------------------------------------

Priority reason:
Puzzle creation is intentionally last. It should build on the final shared rule
contract and solver statuses rather than introduce a third parser or imply that a
budget cutoff means a user puzzle is impossible.

49. Add a browser puzzle editor.
    Current status: Backlog.
   Finish gate:
   - Support create, resize, paint/erase, typed box/goal labels, robot placement,
     undo/redo, clear, and test-play using the existing game-state rules.
   - Make mouse, touch, and keyboard editing accessible, preserve work locally, and
     warn before destructive changes.
   - Reuse the canonical parser and renderer; do not create editor-only Sokoban
     semantics.

50. Define canonical puzzle import, export, and sharing.
    Current status: Partial.
    Existing foundation:
    - Python can load UTF-8 text boards and shared conformance JSON defines the
      current symbol rules.
   Finish gate:
   - Specify a versioned interchange format plus lossless plain-text import/export
     for board rows, typed labels, optional title/author, and future-safe metadata.
   - Use one validation fixture set across browser, Python, editor, benchmarks, and
     documentation; reject unsupported versions and malformed metadata clearly.
   - Add copy/download/upload flows and stable board-content hashes without

51. Add creator validation and solver-assisted publishing checks.
    Current status: Backlog.
   Finish gate:
   - Report structural errors immediately: robot count, box/goal label counts,
