Sokomind improvements and implementation status
===============================================

Status reviewed against the repository after Improvement Sprint 2: conservative
closed-diagonal hard pruning in both solver runtimes.

Standing maintenance rules
--------------------------

- After every change to solver behavior, search strategy, pruning, state
  representation, worker protocol, or solver orchestration, increment the
  solver version ID and keep every browser cache-busting reference synchronized.
- At the end of every sprint, review this file against the repository, update
  the sprint/status heading, move completed work to the appropriate section,
  and record any newly discovered limitations or follow-up work.

The original item numbers and priority order are retained for traceability.
"Done" means the stated outcome is implemented and adequately validated.
"Strong, bounded" means the implementation is useful and well tested within a
deliberate scope, but does not yet provide the complete capability originally
described. "Partial" means meaningful foundations exist, but important parts of
the stated outcome remain. Items in the final section have not been implemented
as backlog capabilities; incidental overlap does not count as delivery.

STRONG IMPLEMENTATION, WITH BOUNDED LIMITATIONS
-----------------------------------------------

3. Add exhaustive differential tests for every hard pruning rule.
   Implemented:
   - A reference BFS checks every current hard pruning rule independently over
     reachable states, including their combined successor behavior.
   - The harness exhaustively enumerates all 780 box, goal, and robot placements
     across every connected 2x3 floor layout. It independently identifies and
     checks the 60 solvable configurations alongside four larger authored boards.
   - It scans the complete corpus and retains the globally smallest false
     positive by floor size, box count, board area, and state size for reporting.
   Bounded limitations:
   - Exhaustive generation does not yet cover larger floors, multiple boxes, or
     typed-label combinations.
   - Counterexamples are selected from the corpus but not structurally minimized.
   - Coverage must continue to be extended whenever hard rejection rules change.
   Remaining validation expansion:
   - Add exhaustive or systematically sampled 3x3 and 3x4 floor layouts.
   - Cover multiple typed boxes, interacting doorways, corrals, and closed
     diagonals as those pruning rules are introduced.
   - Add property-based state generation and automatic counterexample shrinking
     so failures produce the smallest practical reproducible puzzle.

5. Add hidden or generated benchmark families.
   Implemented:
   - Generated mirror, rotation, label permutation, premature-goal, typed
     doorway-import, exact room-packing, and corral-reopening cases.
   - A separate procedural lane-warehouse family covers three and four boxes,
     longer push distances, and distinct typed boxes without deriving its boards
     from the built-in level catalog or the authored strategic seeds.
   - An independent exact A* over canonical push states certifies every procedural
     multi-box case with an admissible label-aware assignment bound.
   - The replay-validated validation suite now contains 16 deterministic cases.
   Bounded limitations:
   - The procedural family currently uses open lane geometry rather than shared
     bottlenecks, interacting staging capacity, or coupled packing orders.
   - The suite remains repository-visible and deterministic, not a secret test
     corpus; more separately maintained challenge layouts would further reduce
     tuning against familiar cases.

7. Add detailed solver performance instrumentation.
   Implemented:
   - Counters and timings cover reachability, matching, hashing, successor
     generation, pruning, caches, frontier behavior, and strategic reasoning.
   - Capability-detected heap telemetry records current, peak, delta, and sample
     counts. Node benchmarks inject a reliable heap sampler; unsupported browser
     workers report that status explicitly instead of returning false zeroes.
   - Isolated benchmark cases record worker loading, search, validation, first
     output, result delivery, shutdown, total process time, and heap deltas.
   - Browser portfolio logs expose first-message, wall-clock, termination-call,
     heap, and cancelled-worker measurements.
   Bounded limitations:
   - Per-allocation profiling and exact GC event/pause counts are not portable to
     ordinary browser workers; the runner reports explicit-GC availability but
     does not force collection during scored searches.
   - The specialized bidirectional director has less lifecycle detail than the
     standard portfolio path, although worker-owned heap telemetry is shared.

8. Replace repeated single-box searches with a compiled push graph.
   Implemented:
   - Static graph nodes encode a box cell and robot-region representative.
   - Transitions and player-aware distances are compiled, cached, and tested
     against the reference implementation.
   Bounded limitations:
   - The graph is geometry-based rather than intrinsically type-aware.
   - It searches forward from box starts instead of reverse-searching from each
     typed goal into reusable typed feasibility tables.

9. Replace string-heavy hot-path state data with dense representations.
   Implemented:
   - Integer cell IDs, dense neighbor/support tables, typed reachability arrays,
     compact parent directions, and deferred walking-path reconstruction.
   - Typed box layouts are encoded as sorted dense cell/label tokens and packed
     into collision-free BigInt identities, cached by immutable box-array object.
   - Exact robot cells or canonical reachable-region IDs are packed into the same
     identity for beam, bounded DFS, IDA*, bidirectional, and baseline search
     transposition maps. Diversity and exact-shard hashing consume the packed
     value directly instead of reconstructing a string.
   - Exhaustive small-board tests verify permutation invariance, typed-layout
     uniqueness, exact robot distinction, and canonical robot-region equivalence.
   Bounded limitations:
   - Coordinate-heavy solver logic still stores boxes as nested JavaScript arrays
     and clones those arrays for successors; the packed form is currently the
     state identity, not the sole state representation.
   - Strategic memo tables and cross-worker IDs retain compact strings where they
     simplify interoperability and diagnostics.
   - Box bitsets and truly incremental move-updated hashing are not implemented.

10. Reuse immutable board analysis across web workers.
    Implemented:
    - Prepared board seeds share topology, push distances, compiled graphs, dense
      board data, and other immutable analysis through structured cloning.
    - Mutable caches remain worker-private.
    Bounded limitations:
    - Structured cloning still duplicates data in worker memory.
    - SharedArrayBuffer is intentionally avoided because the current GitHub Pages
      deployment is not cross-origin isolated.
    - Future pattern and deadlock tables cannot be shared until they exist.

13. Add a goal-commitment safety oracle.
    Implemented:
    - Goal boxes are classified as temporary, conditionally safe, or proven using
      residual perfect matching, room dependencies, static immovability, dynamic
      support demand, typed doorway flow, and bounded exact local analyses.
    - State-complete evidence is cached by robot region and box signature. It can
      downgrade placements that occupy required support or staging cells and can
      prove movable boxes packed inside an exactly completed one-entry room.
    - Beam, bounded DFS, and IDA search treat proven boxes as walls. Forced,
      straight, and sequence macros re-evaluate commitment after every push.
    - The lock is differentially checked over the exhaustive tiny corpus and all
      252 pushes of the known Huge solution; it can be disabled diagnostically.
    Bounded limitations:
    - Movable proof is currently limited to completed bounded rooms; oversized
      rooms and exact corrals cannot promote boxes to proven.
    - Support dependencies are not yet assignment-aware or recursively proven.
    - Baseline push A* and gameplay retain ordinary legal movement semantics.

15. Add cached exact local searches for small rooms and corrals.
    Implemented:
    - Bounded exact room and corral searches are cached by local configuration.
    - They reason about packing, import/export requirements, doorway restoration,
      boundary viability, and reopening robot access.
    Bounded limitations:
    - Exactness applies only inside the selected local abstraction.
    - Rooms, domains, or box sets over conservative size limits are skipped.
    - Some large built-in puzzle regions therefore receive no local exact result.
    - Corral conclusions are currently used conservatively as ordering signals.


PARTIALLY DELIVERED
-------------------

12. Bring the Python/local solver closer to the JavaScript solver.
    Delivered so far:
    - Exact distinct box-to-goal matching, including larger assignments.
    - Bounded heuristic caching.
    - More compact robot-reachability parents and reduced path copying.
    - Replay-safe globally forced push chains collapse into one search edge in
      every push-level search while preserving all GUI movement steps.
    - A deterministic push-beam search has explicit frontier, push-depth, state,
      cancellation, and wall-clock limits. It is exposed through the CLI and
      desktop GUI, and is now the opening phase of local Ultimate Search before
      weighted and exact push A* fallbacks.
    - Conservative mutually frozen box-component detection now complements the
      existing static and 2x2 rules in every local search. It requires that no
      component member have both a free destination and free support square.
    - An independent mechanical-move graph exhaustively checks all generated 2x3
      two-box layouts and confirms that the freeze rule and combined Python
      neighbor pruning retain every transition leading to a solvable state.
    Remaining:
    - Align more Ultimate Search phases and telemetry with the browser director.
    - Port selected topology, sequence macros, closed-diagonal and broader
      deadlocks, IDA, commitment, dependency, room, corral, and doorway capabilities.
    - Remove or integrate remaining redundant priority/heuristic state machinery.

14. Build dynamic support-square dependency graphs.
    Delivered so far:
    - Required standing squares, routes, blockers, support demand, and access loss
      influence move ordering and are cached by state.
    Remaining:
    - Make dependencies assignment-aware so boxes do not independently select the
      same nearest goal.
    - Represent recursive prerequisite box moves and dependency chains.
    - Track recent enabling actions and explicit staging-side requirements.
    - Use contradictions for pruning only after they are proven and differentially
      validated.

16. Model typed doorway flow and staging capacity.
    Delivered so far:
    - Per-room label imports and exports, empty staging capacity, lane readiness,
      diagnostics, penalties, and ordering deltas are computed.
    Remaining:
    - Model typed crossing sequences and time/order-dependent staging occupancy.
    - Reason about particular boxes, multiple rooms, shared corridors, simultaneous
      transfers, and restoration schedules.
    - Confirm suspected contradictions with exact local reasoning.
    - Promote only contradictions confirmed by exact reasoning into safe pruning;
      current flow contradictions remain diagnostic and ordering-only.

18. Add closed-diagonal deadlock detection and more complete freeze detection.
    Delivered so far:
    - Browser and Python searches reject a conservative two-row wall-ended
      diagonal when each open diagonal square is flanked by exactly one box and
      one wall, both boxes face outward toward their wall endpoint, and no square
      in the pattern offers a goal escape.
    - The detector is board-derived and contains no level names, coordinates, or
      saved-solution knowledge.
    - Independent unpruned state graphs enumerate every two-box/two-wall border
      arrangement and both mirror orientations. Only the two outward-facing cases
      are accepted, and both are exhaustively proven unsolvable.
    - The generated family exposed two tempting but solvable false-positive forms
      during development; they remain rejected by the final detector. A goal-escape
      variant is also proven solvable and participates in the combined hard-pruning
      differential suite.
    - The known Huge replay, generated validation families, and all existing hard
      pruning differentials remain intact under solver build 2026-07-22.13.
    Remaining:
    - Recognize box-ended diagonals and safe goal/wall sequences without losing
      valid inward-opening pushes.
    - Add broader transformed, typed, multi-box, and generated diagonal families.
    - Extend recursive freeze detection beyond the current conservative local
      component rule, promoting new cases only after differential validation.

32. Complete project and release hygiene.
    Delivered so far:
    - Added concise architecture documentation defining the general-purpose search
      rule, runtime boundaries, and the proper diagnostic role of saved solutions.
    Remaining:
    - Add a LICENSE, CONTRIBUTING.md, and a changelog or release-note convention.
    - Keep the authoritative implementation and browser/Python capability
      differences explicit as parity work continues.
